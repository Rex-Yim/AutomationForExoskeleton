\section{Experimental Outputs Used in This Chapter}
The final results are generated from the committed MATLAB evaluation scripts rather than from the earlier interim write-up. Public-dataset experiments use sliding windows of 1.0~s with 0.5~s step size at 100~Hz.

Binary classification uses five-fold out-of-fold (OOF) predictions (\texttt{EvaluateSvmConfusion.m}, \texttt{RunSvmDatasetAblation.m}); multiclass evaluation uses \texttt{EvaluateMulticlassConfusion.m}. Percentages match \texttt{generated\_metrics.tex}, exported from \texttt{results/*.mat}.

In parallel with public-dataset training, a three-phone acquisition prototype was completed for simulation-oriented testing. Phones mounted at the lower back and hips provide CSV recordings that can be replayed by the real-time MATLAB pipeline. This prototype is sufficient to validate signal flow, Kalman fusion, and assist-command generation, but it is not yet a substitute for instrumented human-subject trials under load.

Two observations stand out. First, the binary target of exceeding 90\% accuracy is achieved for USC-HAD only (\BinaryOofAccUsc\%) and for the merged setting (\BinaryOofAccMerged\%). Second, HuGaDB-only training is lower at \BinaryOofAccHuGa\%, indicating that cross-session variability and label ambiguity in that dataset make the classification problem harder.

The USC-HAD-only model gives the highest numerical accuracy, but it is not the best deployment candidate for this project because it is trained from a narrower sensing configuration. The merged model is therefore the more meaningful default baseline: its accuracy is lower than the single-dataset USC-HAD result, yet it generalizes across a broader mixture of sensing layouts and activity definitions and is the model copied to \texttt{Binary\_SVM\_Model.mat} for the real-time pipeline.

Figure~\ref{fig:cm} shows the confusion matrix for the merged model. In practical terms, this figure matters more than the scalar accuracy alone because it describes the balance between missed locomotion events and false activations of assistance. For an exoskeleton controller, both error types matter: false negatives can delay assistance when the user starts moving, while false positives can produce unwanted assist engagement during non-locomotion tasks. The merged result is strong enough to support binary assist gating in simulation, but it should still be treated as a baseline rather than a final safety-validated controller.

\section{Discussion of Dataset Effects}
The difference between the three binary accuracies is consistent with the structure of the implemented pipeline. HuGaDB windows can use six IMU streams, whereas USC-HAD windows provide one IMU and are zero-padded to match the 30-D input dimension. As a result, merged training improves diversity but also introduces a domain gap between feature layouts, sampling conditions, and label semantics. The reduction from \BinaryOofAccUsc\% (USC-HAD only) to \BinaryOofAccMerged\% (merged) is therefore not a regression in project value; instead, it is evidence that the final repository evaluates a more realistic and more difficult generalization setting than the interim report did.

Another important point is that the binary task itself is intentionally simplified. The positive class is ``locomotion-like'' rather than a specific gait mode, and HuGaDB window labels are assigned by majority vote over sample-level annotations. This design is appropriate for an assist on/off controller, but it limits how strongly the reported accuracy can be interpreted as full intention understanding. The present results therefore support robust \emph{gating} of assistance, not full activity forecasting or terrain-aware torque generation.

\section{Multiclass Activity Recognition}
Beyond the binary controller, the repository also includes a 12-class ECOC SVM baseline. Its OOF accuracy is \MulticlassOofAcc\%, as shown in Figure~\ref{fig:cm_mc}. This result is substantially lower than the binary accuracy, which is expected: the classifier must separate a larger set of activities spanning walking-related motions, postural states, and dataset-specific behaviors under a shared label space.

From a system-design perspective, the multiclass result should be interpreted as an exploratory baseline rather than as a deployable recognition layer. It demonstrates that the project has progressed beyond a simple binary detector and now has a common label registry and a reproducible multiclass training path. However, the current accuracy is not yet sufficient for fine-grained activity-dependent torque scheduling. Additional class balancing, subject-wise validation, and phone-specific data collection would be required before multiclass predictions could be trusted for direct control decisions.

\section{Real-Time Pipeline Behavior}
Figure~\ref{fig:pipeline} presents the output of the online MATLAB pipeline, combining Kalman-based hip-angle estimation with SVM classification and finite-state-machine (FSM) smoothing. The figure shows that the pipeline produces a stable latched assist command rather than rapidly toggling in response to individual window predictions.

\begin{figure}[htbp]
  \centering
  \includegraphicsresult{pipeline_output.png}
  \caption{Real-time pipeline output: hip flexion angle (top), control command versus acceleration magnitude (middle), and annotation panel when available (bottom).}
  \label{fig:pipeline}
\end{figure}

This behavior is significant because the FSM design directly addresses a practical control issue: a high-accuracy classifier can still generate undesirable actuator chatter if predictions fluctuate near transition boundaries. Requiring three consecutive locomotion predictions to enter assist and five consecutive non-locomotion predictions to exit assist makes the controller conservative in a useful way. The trade-off is extra response delay, so the original 100--200~ms predictive-horizon claim should now be regarded as a design aspiration rather than a re-validated result of the submitted repository.

\section{Sequence-Model Status}

It is intentionally not treated as the primary reported result in this chapter because its current evaluation uses a stratified 20\% holdout rather than the five-fold OOF protocol used for the SVM baselines. Until both models are evaluated under the same validation setup, a direct performance comparison would not be methodologically fair.

\section{Overall Interpretation}
Taken together, the results show that the project has successfully delivered a reproducible software baseline for exoskeleton assist gating. The binary SVM pipeline is mature enough to support simulation-level decision making with strong accuracy on public data, the Kalman and FSM blocks are integrated into an end-to-end control flow, and the report metrics are now traceable to versioned MATLAB outputs. At the same time, the discussion also makes clear what remains unfinished: subject-specific tuning, truly predictive transition detection, hardware deployment, and controlled user validation under realistic load-transport tasks.
