\posterbox{How to read the percentages}{
  Every percentage on this poster is a \textbf{classification accuracy}: the share of IMU windows whose predicted label matches the dataset ground truth. For example, \textbf{95.89\%} means about \textbf{96 out of every 100 windows} were labeled correctly.

  \vspace{0.06em}
  \begin{compactitem}
    \item \textbf{5-fold OOF} means five-fold out-of-fold cross-validation: the model is repeatedly trained on 80\% of the data and tested on the remaining 20\%, so every window is evaluated once while unseen during its own test fold.
    \item \textbf{20\% holdout} means one fixed split: 80\% of the windows are used for training and 20\% are kept aside for a final test.
    \item \textbf{Binary} percentages measure correct \textit{active vs.\ inactive} decisions; \textbf{multiclass} percentages measure correct assignment to the dataset's native activity class.
  \end{compactitem}
}

\noindent
\begin{minipage}[t]{0.485\textwidth}
  \posterbox{Problem and approach}{
    \begin{compactitem}
      \item Manual load handling above 20~kg raises fatigue and musculoskeletal injury risk.
      \item An exoskeleton is useful when assistance engages and releases at the right time.
      \item This work targets a \textbf{software supervisor}: infer active movement from IMUs and issue a reliable assist on/off signal without EMG.
      \item The goal is not full torque control yet; it is a robust decision layer that can sit above future hardware control.
    \end{compactitem}
  }

  \posterbox{Implemented workflow}{
    \begin{compactitem}
      \item \textbf{Input:} canonical GitHub v1 raw under \texttt{v1\_cleanup\_github/} for training and held-out replay for simulation; a second public benchmark provides comparison under the same feature layout.
      \item \textbf{Dataset handling:} loader applies the official gyro-corruption rules at read time, distinguishes \texttt{single\_activity} from \texttt{multi\_activity\_sequence} sessions, builds \texttt{hugadb\_dataset.mat}, and records quality metadata so invalid sessions/windows are skipped rather than silently used.
      \item \textbf{Session variation coverage:} controlled single-activity recordings include \texttt{walking}, \texttt{running}, \texttt{sitting}, \texttt{standing}, \texttt{bicycling}, \texttt{sitting\_in\_car}, and \texttt{down\_by\_elevator}; mixed \texttt{various} recordings are treated as the streaming-style multi-activity sequences.
      \item \textbf{Fusion:} Kalman filtering is optional and used for a replay-time segment-pitch visualization trace, not for classifier preprocessing.
      \item \textbf{Windowing:} 1.0~s windows at 100~Hz with 50\% overlap.
      \item \textbf{Features:} \textbf{30-D} handcrafted vector, five statistics per IMU slot across six slots.
      \item \textbf{Primary model:} RBF SVM for \textbf{binary gating}: active-movement windows vs.\ inactive windows, using dataset-specific activity-to-binary maps.
      \item \textbf{Supervisory smoothing:} FSM hysteresis, 3 active predictions to engage assist and 5 inactive predictions to release.
    \end{compactitem}
  }
\end{minipage}\hfill

  \posterbox{Data and evaluation protocol}{
    \begin{compactitem}
      \item The final pipeline uses six on-body IMU streams per window, full native class coverage, and \textbf{multi-activity sequence sessions} as the realistic streaming policy; single-activity sessions are controlled subsets (walking, running, sitting, standing, bicycling, sitting-in-car, elevator descent).
      \item The canonical raw layout is \texttt{v1\_cleanup\_github/} (flat \texttt{HuGaDB\_v1\_*.txt}); the loader threads optional manifest metadata into the cache for validation and rebuild traceability.
      \item The second benchmark uses one IMU stream under the same 30-D layout for cross-dataset comparison.
      \item The \textbf{binary SVM} is the main baseline; the active checkpoint is trained under the held-out-subject policy used by replay evaluation.
      \item \textbf{Binary} asks a simple control question: should assist stay \textbf{off} or turn \textbf{on}? \textbf{Multiclass} asks a harder recognition question: which exact activity is occurring?
      \item \textbf{Multiclass SVM} (12 classes per benchmark) and \textbf{binary/multiclass LSTM} are supporting baselines; the SVM uses stratified subsamples and the LSTMs use separate stratified 20\% holdouts, so neither is a direct deployment comparison.
    \end{compactitem}
  }

\begin{minipage}[t]{0.485\textwidth}
  \posterbox{How to read the supporting baselines}{
    \begin{compactitem}
      \item The pipeline plot shows why the FSM matters: it turns noisy window-level predictions into a smoother and more usable assist signal.
      \item The \textbf{binary} task is easier because several activities are merged into two groups; the \textbf{multiclass} task is harder because it must separate the exact activity label.
      \item Multiclass SVM OOF scores (\textbf{\MulticlassOofAccUsc\%} / \textbf{\MulticlassOofAccHuGa\%}) are informative for analysis, but not yet strong enough for fine-grained control.
      \item \ifnum\HasMulticlassLstmMetrics=1 Multiclass LSTM holdout (\textbf{\LstmMulticlassHoldoutAccUsc\%} / \textbf{\LstmMulticlassHoldoutAccHuGa\%}) is a summary metric across distinct native label spaces.\else Multiclass LSTM metrics are optional and are omitted until the per-dataset sequence models are trained.\fi
    \end{compactitem}
  }

  \posterbox{Current limits}{
    \begin{compactitem}
      \item No hardware-in-the-loop or human-subject load trials are reported here.
      \item End-to-end latency still needs direct benchmarking during future hardware-integrated validation.
      \item Current output is \textbf{assist gating}, not full torque scheduling or terrain-aware control.
      \item Multiclass and LSTM baselines still need a stronger evaluation protocol aligned with the main binary SVM path.
    \end{compactitem}
  }
\end{minipage}\hfill
\begin{minipage}[t]{0.485\textwidth}
  \posterbox{Next steps}{
    \begin{compactitem}
      \item Align multiclass and LSTM evaluation with the same level of rigor used for the main binary SVM baseline.
      \item Add hardware integration, latency measurement, and load-carrying trials.
      \item Move from binary assist gating toward subject-specific, activity-aware assistance.
    \end{compactitem}

  \posterbox{Summary}{
    \begin{compactitem}
      \item \textbf{Delivered:} canonical GitHub v1 data flow, protocol-aware binary SVM, optional Kalman visualization, FSM smoothing, reproducible metrics, and replay in MATLAB.
      \item \textbf{Best validated result:} binary SVM \textbf{\BinaryOofAccHuGa\%} five-fold OOF on the streaming split.
      \item \textbf{System role:} simulation-level assist gating supervisor.
      \item \textbf{Value:} a credible, traceable software baseline that separates benchmark comparisons from the final deployed method.
    \end{compactitem}
  }
